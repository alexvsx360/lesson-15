מה זה Git? 📦

Git הוא מערכת ניהול גרסאות (Version Control System) - כלי שמאפשר לעקוב אחר שינויים בקוד ולשמור היסטוריה של כל השינויים.

למה Git חשוב?

  • היסטוריה: רואים את כל השינויים שעשיתם
  • גיבוי: הקוד שמור - לא מאבדים עבודה
  • שיתוף: קל לשתף קוד עם אחרים
  • שיתוף פעולה: כמה אנשים יכולים לעבוד על אותו פרויקט
  • חזרה אחורה: אפשר לחזור לגרסה קודמת

דוגמה - בלי Git:

# קבצים ישנים - מבלבל!
project-v1.js
project-v2.js
project-v2-final.js
project-v2-final-really.js
project-v3.js
# איזה קובץ הוא הנוכחי? מה השתנה? 😕

עם Git:

# קובץ אחד - Git שומר את ההיסטוריה
project.js
# Git יודע בדיוק מה השתנה ומתי! ✅

התקנה והגדרה ⚙️

1. התקנת Git:

Windows:

הורידו מ-git-scm.com והתקינו

Mac:

brew install git

Linux:

sudo apt-get install git

2. הגדרת זהות:

# הגדרת שם
git config --global user.name "דני כהן"

# הגדרת אימייל
git config --global user.email "danny@example.com"

# בדיקה
git config --global --list

3. בדיקה שההתקנה עבדה:

git --version
# Git version 2.40.0
חשוב: הגדרת זהות פעם אחת - Git ישתמש בזה לכל ה-commits שלכם.

Repository - מה זה? 📁

Repository (או בקיצור "repo") הוא תיקיית פרויקט שמנוהלת על ידי Git. Git עוקב אחר כל הקבצים בתיקייה הזו.

יצירת Repository חדש:

# נווט לתיקיית הפרויקט
cd my-project

# אתחל Git repository
git init

# Git יצר תיקייה נסתרת .git
# עכשיו Git עוקב אחר כל הקבצים כאן!

מה קורה ב-git init?

  • נוצרת תיקייה נסתרת .git
  • Git מתחיל לעקוב אחר הקבצים בתיקייה
  • התיקייה הופכת ל-Repository
  • עכשיו אפשר להשתמש בפקודות Git

⚠️ זהירות:

אל תמחקו את התיקייה .git! היא מכילה את כל ההיסטוריה של הפרויקט.

פקודות בסיסיות - git add 📝

git add מוסיף קבצים ל-Staging Area - "אזור ההכנה" לפני ה-commit.

שלושה שלבים:

1. Working Directory
(קבצים שערכתם)
↓ git add
2. Staging Area
(קבצים מוכנים ל-commit)
↓ git commit
3. Repository
(commits שמורים)

דוגמאות:

# הוסף קובץ אחד
git add index.html

# הוסף כל הקבצים בתיקייה
git add .

# הוסף כל הקבצים עם סיומת מסוימת
git add *.js

# הוסף תיקייה שלמה
git add src/

בדיקת סטטוס:

git status
# מציג אילו קבצים שונו, אילו ב-staging, וכו'

git commit - שמירת שינויים 💾

git commit שומר את השינויים ב-Staging Area כ-commit - "צילום" של המצב הנוכחי.

יצירת commit:

# Commit עם הודעה
git commit -m "הוספתי כפתור התחברות"

# הודעה מפורטת
git commit -m "הוספתי כפתור התחברות

- הוספתי HTML לכפתור
- הוספתי CSS לעיצוב
- הוספתי JavaScript לפונקציונליות"

עקרון חשוב:

Commit קטן ומובן

  • כל commit צריך להיות שינוי אחד לוגי
  • הודעה ברורה - מה השתנה ולמה
  • דוגמאות טובות: "תיקנתי באג בחישוב מחיר", "הוספתי תמיכה ב-RTL"
  • דוגמאות רעות: "שינויים", "עדכון", "fix"

תהליך מלא:

# 1. ערוך קבצים
# 2. בדוק מה השתנה
git status

# 3. הוסף staging
git add .

# 4. Commit
git commit -m "הוספתי תכונה חדשה"

# 5. בדוק את ההיסטוריה
git log

git status & log - מידע על הפרויקט 📊

git status - מצב נוכחי:

git status

# פלט לדוגמה:
# On branch main
# Changes not staged for commit:
#   modified:   index.html
#   modified:   style.css
# 
# Untracked files:
#   new-file.js
# 
# Changes to be committed:
#   modified:   script.js

git log - היסטוריית commits:

# היסטוריה מלאה
git log

# היסטוריה קצרה (שורה אחת לכל commit)
git log --oneline

# היסטוריה עם גרף
git log --oneline --graph

# רק 5 האחרונים
git log -5

git diff - מה השתנה?

# מה השתנה בקבצים שעדיין לא ב-staging
git diff

# מה השתנה בקבצים ב-staging
git diff --staged

# מה השתנה בקובץ מסוים
git diff index.html

.gitignore - קבצים להתעלם מהם 🚫

.gitignore הוא קובץ שאומר ל-Git אילו קבצים לא לעקוב אחריהם.

למה צריך .gitignore?

  • node_modules/ - תיקייה גדולה, לא צריך לשמור
  • קבצי build - נוצרים אוטומטית
  • קבצי סביבה - .env עם סיסמאות
  • קבצי IDE - .vscode, .idea
  • קבצים זמניים - .log, .tmp

דוגמה ל-.gitignore:

# node_modules
node_modules/

# קבצי סביבה
.env
.env.local

# קבצי build
dist/
build/

# קבצי IDE
.vscode/
.idea/

# קבצים זמניים
*.log
*.tmp
.DS_Store

# אבל שמור את זה:
!important.log

יצירת .gitignore:

# צור קובץ .gitignore
touch .gitignore

# או ערוך אותו
nano .gitignore
# הוסף את הקבצים/תיקיות להתעלם מהם
טיפ: יש templates מוכנים ל-.gitignore לפי סוג הפרויקט (Node.js, Python, וכו') ב-github.com/github/gitignore

מה זה GitHub? 🌐

GitHub הוא שירות אירוח (hosting) ל-Git repositories. זה כמו "Google Drive" לקוד - מקום לשמור ולשתף את הקוד שלכם.

Git

  • כלי מקומי
  • על המחשב שלכם
  • ניהול גרסאות
  • פקודות command line

GitHub

  • שירות מקוון
  • בענן
  • אירוח ושיתוף
  • ממשק web + Git

יתרונות GitHub:

  • גיבוי בענן: הקוד שמור גם אם המחשב נשבר
  • שיתוף קל: קל לשתף קוד עם אחרים
  • שיתוף פעולה: כמה אנשים יכולים לעבוד יחד
  • Portfolio: מציג את הפרויקטים שלכם
  • Issues & PRs: כלים לניהול פרויקטים

💡 חלופות ל-GitHub:

  • GitLab: דומה ל-GitHub, עם יותר תכונות
  • Bitbucket: שירות נוסף
  • GitHub Enterprise: גרסה ארגונית

יצירת Repository ב-GitHub 🆕

שלבים:

  1. היכנסו ל-github.com
  2. הירשמו/התחברו
  3. לחצו על "+" → "New repository"
  4. מלאו פרטים:
    • Repository name: שם הפרויקט
    • Description: תיאור קצר
    • Public/Private: ציבורי או פרטי
    • Initialize with README: אפשר לסמן
  5. לחצו "Create repository"

חיבור Repository מקומי ל-GitHub:

# אם כבר יש לכם repository מקומי
git remote add origin https://github.com/username/repo-name.git

# בדיקה
git remote -v

# שליחה ל-GitHub
git push -u origin main

או - יצירה חדשה:

# צרו repository חדש
git init
git add .
git commit -m "Initial commit"

# חיבור ל-GitHub
git remote add origin https://github.com/username/repo-name.git
git branch -M main
git push -u origin main

git push & pull - שליחה וקבלה 📤📥

git push - שליחה ל-GitHub:

# שליחת commits ל-GitHub
git push

# בפעם הראשונה - ציין branch
git push -u origin main

# אחר כך אפשר פשוט:
git push

git pull - קבלה מ-GitHub:

# משיכת שינויים מ-GitHub
git pull

# זה בעצם:
# git fetch (מביא שינויים)
# + git merge (ממזג אותם)

תרחיש נפוץ:

# עבדתם על המחשב בבית
git add .
git commit -m "הוספתי תכונה חדשה"
git push

# עכשיו במחשב אחר (עבודה)
git pull  # מביא את השינויים מהבית

# עבדתם בעבודה
git add .
git commit -m "תיקנתי באג"
git push

# חזרתם הביתה
git pull  # מביא את השינויים מהעבודה

⚠️ זהירות - Conflicts:

אם שני אנשים שינו את אותו קובץ, Git יבקש מכם לפתור את ה-conflict. זה נושא מתקדם יותר.

git clone - העתקת Repository 📋

git clone מעתיק repository מ-GitHub (או ממקום אחר) למחשב שלכם.

שימוש:

# העתק repository
git clone https://github.com/username/repo-name.git

# עם שם תיקייה מותאם
git clone https://github.com/username/repo-name.git my-project

# זה יוצר תיקייה חדשה עם כל הקוד!

מתי משתמשים?

  • התחלת עבודה על פרויקט: מישהו אחר יצר repository
  • העתקת פרויקט: רוצים לעבוד עליו מקומית
  • Fork: רוצים לעבוד על פרויקט של מישהו אחר
  • תיקון באגים: רוצים לעזור בפרויקט קוד פתוח

דוגמה מעשית:

# מצאתם פרויקט מעניין ב-GitHub
# לחצתם על "Code" → העתקתם את ה-URL

# העתקה
git clone https://github.com/username/awesome-project.git

# נכנסים לתיקייה
cd awesome-project

# עכשיו יש לכם את כל הקוד מקומית!

Branches - ענפים 🌿

Branch (ענף) הוא "עותק" של הקוד שבו אפשר לעבוד בלי להשפיע על הקוד הראשי.

למה צריך Branches?

  • עבודה על תכונה חדשה: לא רוצים לשבור את הקוד הקיים
  • תיקון באגים: עובדים על תיקון בנפרד
  • ניסויים: מנסים משהו חדש בלי סיכון
  • שיתוף פעולה: כל אחד עובד על branch שלו

יצירת Branch:

# צור branch חדש
git branch feature-login

# עבור ל-branch
git checkout feature-login

# או - צור ועבור בבת אחת
git checkout -b feature-login

# ב-Git 2.23+ אפשר גם:
git switch -c feature-login

רשימת Branches:

# רשימת כל ה-branches
git branch

# עם * על ה-branch הנוכחי
# * main
#   feature-login
#   bugfix-header

git merge - מיזוג ענפים 🔀

git merge ממזג שינויים מ-branch אחד ל-branch אחר.

תהליך מיזוג:

# 1. עבדתם על feature branch
git checkout feature-login
# ... עבדתם, עשיתם commits ...

# 2. חזרו ל-main
git checkout main

# 3. מיזוג ה-feature ל-main
git merge feature-login

# 4. שליחה ל-GitHub
git push

דוגמה מעשית:

# התחלתם לעבוד על תכונה חדשה
git checkout -b add-dark-mode

# עבדתם על התכונה
git add .
git commit -m "הוספתי מצב כהה"
git commit -m "תיקנתי באגים במצב כהה"

# התכונה מוכנה - מיזוג ל-main
git checkout main
git merge add-dark-mode

# אפשר למחוק את ה-branch (לא חובה)
git branch -d add-dark-mode

💡 Best Practice:

  • עבדו על feature branch, לא על main
  • מיזגו ל-main רק כשהתכונה מוכנה
  • מחקו branches ישנים אחרי מיזוג
  • שמרו על main נקי ויציב

Pull Requests - בקשות מיזוג 🔄

Pull Request (PR) הוא דרך לבקש למזג שינויים מ-branch אחד ל-branch אחר ב-GitHub. זה מאפשר לבדוק ולשוחח על השינויים לפני המיזוג.

תהליך Pull Request:

  1. עבדתם על branch: יצרתם feature branch ועשיתם commits
  2. שלחתם ל-GitHub: git push
  3. יצירת PR: ב-GitHub, לחצו "New Pull Request"
  4. בחירת branches: מ-feature ל-main
  5. תיאור: הסבירו מה השתנה ולמה
  6. ביקורת: אחרים יכולים לבדוק ולהוסיף הערות
  7. מיזוג: אחרי אישור - "Merge Pull Request"

יתרונות Pull Requests:

  • ביקורת קוד: אחרים יכולים לבדוק את השינויים
  • דיון: אפשר לדון על השינויים
  • בדיקות: אפשר להריץ בדיקות אוטומטיות
  • היסטוריה: רואים למה שינויים נעשו
  • למידה: לומדים מקוד של אחרים

דוגמה - PR טוב:

## מה השתנה?
הוספתי מצב כהה לאפליקציה

## למה?
משתמשים ביקשו תמיכה במצב כהה

## מה נבדק?
- [x] עובד על Chrome
- [x] עובד על Firefox
- [x] נבדק על מובייל

## תמונות
[תמונות של המצב הכהה]

שאלות כיתה – Git + GitHub 🧠

1. מה ההבדל בין Git ל-GitHub? מתי משתמשים בכל אחד?

Git: כלי מקומי לניהול גרסאות - רץ על המחשב שלכם. משתמשים בו לניהול גרסאות מקומית - commits, branches, history, וכו'.

GitHub: שירות מקוון לאירוח Git repositories. משתמשים בו לשמירת קוד בענן, שיתוף עם אחרים, שיתוף פעולה, Pull Requests, Issues, וכו'.

מתי משתמשים: Git - כל פעם שעובדים על קוד (מקומית). GitHub - כשצריך לשתף, לגבות בענן, או לעבוד בצוות. בדרך כלל משתמשים בשניהם יחד - Git לניהול מקומי, GitHub לאירוח ושיתוף.

2. מה ההבדל בין git add, git commit, ו-git push? מה כל אחד עושה?

git add: מוסיף קבצים ל-Staging Area - "מכין" אותם ל-commit. זה לא שומר את השינויים, רק מסמן אותם כמוכנים.

git commit: שומר את השינויים ב-Staging Area כ-commit - "צילום" של המצב הנוכחי. זה יוצר snapshot שמור ב-repository המקומי.

git push: שולח את ה-commits מ-repository מקומי ל-GitHub (או שרת אחר). זה מעלה את הקוד לענן.

אנלוגיה: git add = הכנת חבילה, git commit = חתימה על החבילה, git push = שליחת החבילה בדואר.

3. מה זה Branch ולמה צריך אותו? מתי יוצרים branch חדש?

Branch: "עותק" של הקוד שבו אפשר לעבוד בלי להשפיע על הקוד הראשי (main). זה כמו יצירת עותק של מסמך לעריכה.

למה צריך: כדי לעבוד על תכונות חדשות, תיקוני באגים, או ניסויים בלי לשבור את הקוד הקיים. זה מאפשר לעבוד בצוות - כל אחד על branch שלו.

מתי יוצרים: כשמתחילים לעבוד על תכונה חדשה, תיקון באג, או ניסוי. הרעיון הוא לשמור את main נקי ויציב, ולעבוד על branches נפרדים. אחרי שהעבודה מוכנה, ממזגים את ה-branch ל-main.

4. מה זה Pull Request? מה התהליך המלא של יצירת PR?

Pull Request (PR): בקשה למזג שינויים מ-branch אחד ל-branch אחר. זה מאפשר לבדוק ולדון על השינויים לפני המיזוג.

תהליך מלא:
1. עובדים על feature branch (לא על main)
2. עושים commits
3. שולחים ל-GitHub: git push
4. ב-GitHub: לחצו "New Pull Request"
5. בחרו branches: מ-feature ל-main
6. כתבו תיאור: מה השתנה ולמה
7. בקשו review: אחרים יכולים לבדוק ולהוסיף הערות
8. אחרי אישור: "Merge Pull Request"
9. מחיקת branch (לא חובה)

יתרונות: ביקורת קוד, דיון, בדיקות אוטומטיות, היסטוריה, למידה.

5. מה זה .gitignore ולמה חשוב להשתמש בו? תן דוגמאות לקבצים שצריך להתעלם מהם.

.gitignore: קובץ שאומר ל-Git אילו קבצים/תיקיות לא לעקוב אחריהם - להתעלם מהם.

למה חשוב: יש קבצים שלא רוצים לשמור ב-Git - node_modules (גדול מדי), קבצי סביבה (.env עם סיסמאות), קבצי build (נוצרים אוטומטית), קבצי IDE, וכו'. בלי .gitignore, כל הקבצים האלה יכנסו ל-repository ויגדילו אותו ויחשפו מידע רגיש.

דוגמאות לקבצים להתעלם:
• node_modules/ - תיקיית dependencies
• .env - קבצי סביבה עם סיסמאות
• dist/, build/ - קבצי build
• .vscode/, .idea/ - קבצי IDE
• *.log, *.tmp - קבצים זמניים
• .DS_Store - קבצי מערכת (Mac)
